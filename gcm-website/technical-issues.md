# 官網技術問題診斷與修復方案

**狀態說明**：因網路政策阻擋，無法直接讀取 guanchengmartech.com 原始碼確認實際
外掛名稱與程式碼位置，以下依症狀描述給出最可能成因與通用修復步驟。實際套用前，
請先在 WP 後台核對外掛清單，或貼上 `檢視網頁原始碼` 的相關片段給我，我可以給出
更精確的修法。務必先用 UpdraftPlus 等外掛做全站備份再動手。

## 問題一：表單「Please enable JavaScript」文字外露

### 最可能成因
這是 **Contact Form 7** 的典型症狀。CF7 的表單標籤裡預設藏了一段
`<noscript>Please enable JavaScript in your browser to complete this form.</noscript>`，
正常情況下瀏覽器啟用 JS 時不會顯示，只有在以下情況才會外露：
1. 頁面的 JS 檔案沒有正確載入（被快取外掛、CDN、或某個效能外掛的「延遲載入
   JS」設定誤殺）
2. 主題或其他外掛與 CF7 的 JS 有衝突，導致 `wpcf7.js` 沒有正確初始化
3. 表單被放進某個用 AJAX／Elementor Popup 動態載入的區塊，JS 綁定時機錯亂

### 修復步驟
1. 開瀏覽器開發者工具（F12）→ Network 分頁，重新整理表單所在頁面，確認
   `contact-form-7/includes/js/index.js`（或類似路徑）是否回傳 200，若是 404
   或被擋，先解決載入問題
2. 檢查是否有效能／快取外掛（如 WP Rocket、Autoptimize、LiteSpeed Cache）把
   CF7 的 JS 排除在「合併/延遲載入」名單外，這些外掛通常有「排除特定 JS 檔案」
   設定，CF7 的 JS 必須排除在延遲載入之外
3. 確認主題沒有覆寫 `wp_footer()` 或提早結束輸出，導致 CF7 的 JS 無法正確掛載
4. 若問題持續，暫時停用其他外掛逐一排查（先備份），確認是否為外掛衝突
5. 修復後用無痕視窗＋清除快取重新測試，確認 `<noscript>` 文字不再以純文字
   形式出現在畫面上（它應該永遠存在於原始碼中，只是啟用 JS 時不該被看到）

## 問題二：右鍵保護外掛「error: Content is protected !!」字串外洩

### 最可能成因
這類「禁止右鍵／禁止複製」外掛（常見如 WP Content Copy Protection、
No Right Click Images Plugin 等）通常透過在頁面注入一段 JavaScript，
攔截 `contextmenu`、`copy`、`selectstart` 事件並跳出提示訊息。字串外洩到
「頁面原始內容」中通常代表：
1. 提示訊息文字被外掛以**明文字串寫死在 HTML／DOM 裡**（例如塞進一個隱藏的
   `<div>` 或 `alert()` 呼叫前的字串常數），搜尋引擎爬蟲會直接讀到這段文字，
   當成頁面內容的一部分索引
2. 該外掛的訊息文字沒有做 i18n 或條件式輸出，導致同一段英文字串出現在所有
   頁面原始碼裡，影響頁面文字的關鍵字密度與語意純淨度

### 修復步驟
1. 到 WP 後台「外掛」清單，找出負責右鍵保護的外掛名稱（若不確定，可用瀏覽器
   檢視原始碼搜尋 `Content is protected` 關鍵字，附近的程式碼註解或檔案路徑
   通常會透露外掛身分）
2. **優先建議：直接停用並移除此類外掛。** 右鍵保護對防止內容被複製幾乎無效
   （任何人都能透過開發者工具或關閉 JS 繞過），卻會犧牲 SEO（外洩字串汙染
   頁面內容）與使用體驗（合法使用者也無法選取文字），這類外掛在 2024 年後
   的 WordPress 生態中已普遍不建議使用
3. 若基於業務理由仍要保留類似保護（例如防止圖片被直接另存新檔），改用更
   輕量的做法：
   - 圖片加浮水印，而非阻擋右鍵
   - 針對圖片元素單獨套用 CSS `pointer-events: none` 搭配透明遮罩，
     不影響文字選取與頁面文字內容
   - 避免使用會把提示文字寫進 DOM 或以 `document.write` 注入字串的外掛版本
4. 移除外掛後，務必重新檢查頁面原始碼（Ctrl+U 或檢視原始碼），確認
   `Content is protected` 字串已完全消失，並清除所有快取（外掛快取、
   CDN 快取、瀏覽器快取）

## 問題三：新增「禁止截圖」功能

### 重要提醒
瀏覽器端網頁**技術上無法真正阻止使用者截圖**，作業系統層級的螢幕截圖
（如 Windows Snipping Tool、macOS Cmd+Shift+4）完全不經過網頁 JS，
任何宣稱「防止截圖」的外掛實際上只能做到「增加一點點麻煩」，無法做到
真正禁止。這點需要在對內溝通時說清楚，避免對這個功能有不切實際的期待，
也不建議在對外文案中宣稱「防截圖」，以免誤導使用者對隱私保護程度的認知。

### 可行的替代做法（依防護目的選擇）
1. **如果目的是保護文字內容不被複製轉貼**：
   用前述問題二的輕量做法（CSS 選取限制、避免外洩提示字串），效果與
   全面禁止截圖類似，但不犧牲 SEO
2. **如果目的是保護機密文件（如報價單、合約 PDF）不被隨意散布**：
   截圖防護在網頁端做不到，應該在**文件本身**做浮水印（浮水印上顯示
   收件人資訊，萬一外流可追溯來源），這是業界對付機密文件外流的標準做法，
   建議套用在 `gcm-vault/05_templates/` 產出的 PDF 交付物上
3. **如果目的是防止頁面被錄影／連續截圖爬取內容**：
   同樣做不到技術上的完全防止，但可以透過浮水印疊加、內容分段延遲載入
   增加難度，效益有限，需評估是否值得犧牲使用體驗

### 建議
不建議實作「禁止截圖」功能本身（技術上做不到，且沿用右鍵保護外掛的老路
會重蹈問題二的 SEO 外洩風險）。如果真正的需求是保護機密文件外流，改用
PDF 浮水印方案，效益更高、風險更低。
